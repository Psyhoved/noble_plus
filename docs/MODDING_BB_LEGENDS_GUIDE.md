# Battle Brothers + Legends Modding Standard (Noble Plus)

> Версия: 2.0  
> Дата: 23.02.2026  
> Статус: **Canonical** (основной технический стандарт проекта)

## 1. Зачем этот документ

Этот документ фиксирует рабочий стандарт разработки модов для Battle Brothers + Legends + MSU/Modern Hooks на основе **внешних источников** и проверок в реальном запуске игры.

Цель: исключить цикл «правим код -> тестируем -> изменения не дошли до игры -> повторяем».

## 3. Базовые технические правила (обязательные)

## 3.1 Загрузка и структура

1. Игра монтирует `data/` как корень виртуальной FS.
2. Автоскан сценариев/событий/амбиций происходит из `data/scripts/...`.
3. `data/mod_<id>/...` не равно `data/scripts/...`: это другой VFS namespace (`mod_<id>/...`).
4. Хуки из `mod_<id>/hooks` должны быть подключены явно (через очередь/регистрацию).

## 3.2 Регистрация мода и порядок

1. Мод обязан объявлять уникальный ID и зависимости (`mod_msu`, `mod_legends`).
2. Любой hook-патч должен выполняться после загрузки зависимостей.
3. При использовании Modern Hooks/MSU придерживаться документированного API и не опираться на недокументированные внутренние поля.

## 3.3 Работа с API (MSU / Hooks)

1. Использовать только документированные части API, если нет необходимости в low-level workaround.
2. Любой workaround по недокументированным API должен иметь:
   - комментарий `WHY`,
   - ссылку на источник/кейс,
   - fallback-поведение.
3. Настройки мода (ModSettings) реализовывать по MSU Wiki-паттерну, а не по предположениям.

## 3.4 Правила для Squirrel-кода

1. `create()` не должен зависеть от `World.*`, если объект мира не гарантирован.
2. В hooks вызывать оригинальный метод через сохранённую ссылку + `bindenv`, где требуется.
3. Не блокировать функционал по «хрупким» string-prefix фильтрам, если есть allowlist/registry.
4. Каждый критичный модуль должен логировать свой bootstrap и режим работы.

## 4. Стандарт разработки в проекте Noble Plus

## 4.1 Canonical source of truth

1. Логика амбиций:  
`mod/mod_noble_plus/scripts/config/noble_plus_ambitions_config.nut`
2. Runtime-регистрация:  
`mod/mod_noble_plus/scripts/ambitions/noble_plus_ambitions_runtime.nut`
3. Документация: этот файл (`docs/MODDING_BB_LEGENDS_GUIDE.md`).

## 4.2 Обязательный workflow (не пропускать)

1. Внести правки в репозитории.
2. Выполнить deploy в live-скрипты игры:
```bash
./tools/deploy_live_scripts.sh
```
3. Проверить, что SHA1 совпали (скрипт делает fail-fast).
4. Запустить игру и проверить логи:
   - `[NoblePlus] deploy check: preload config/runtime includes loaded`
   - `[NoblePlus][Ambitions] initialized: ...`
   - `[NoblePlus][Ambitions] suppressor targets: ...`
5. Только после этого считать игровой тест валидным.

## 4.3 Что считается невалидным тестом

1. Игра запущена без `deploy_live_scripts.sh`.
2. Нет проверки хэшей или есть mismatch.
3. Нет базовых bootstrap-маркеров в `log.html`.

## 5. Критический аудит текущего состояния проекта

## 5.1 Что сделано правильно

1. Введён config-driven подход для амбиций.
2. Добавлен runtime allowlist и safety-net в менеджере амбиций.
3. Добавлен deploy-скрипт с SHA1-проверкой.
4. Улучшена диагностируемость в логах.

## 5.2 Ключевые архитектурные проблемы

1. **Dual-source runtime**: hooks и scripts читаются из разных мест (`mod/mod_noble_plus` vs `data/scripts`), что легко создаёт гибридные версии.
2. **Suppressor brittle surface**: явный список блокируемых ambition-классов всё ещё подвержен пропускам DLC/модовых классов.
3. **Документация исторически отставала от кода**, из-за чего возникали повторные неверные решения.
4. **Низкий барьер для невалидного теста**: до сих пор можно руками запустить игру и сделать ложные выводы.

## 5.3 Риски при текущем состоянии

1. Повторные циклы регрессий «не дошло до игры».
2. Ложные диагнозы причин бага.
3. Потеря времени на исправления несуществующих проблем.

## 6. Предложения по улучшению (приоритет)

### P0 (немедленно)

1. Ввести `tools/preflight_live.sh`:
   - проверка deploy + SHA1 + наличие критичных маркеров в логе.
2. Сделать policy: игровой тест без preflight считается недействительным.
3. Вынести список suppressor-классов в отдельный конфиг и версионировать его.
4. Ввести обязательный “Evidence block” в PR/изменениях docs:
   - источник (URL),
   - уровень достоверности,
   - как проверено в игре.

### P1 (ближайший этап)

1. Автоматический `tools/log_scan.sh` для `log.html` (ошибки/маркеры Noble Plus).
2. Единый формат логов `[NoblePlus][Module][Code]`.
3. Таблица совместимости версий: BB / Legends / MSU / Modern Hooks.

### P2 (следующий горизонт)

1. Мини smoke-test suite через MCP (запуск -> новый старт -> до 2-го дня -> контроль пула амбиций).
2. Шаблоны модулей (`scripts/lib/noble_plus_*`) для типовых паттернов.

## 7. Антипаттерны (запрещено)

1. Делать выводы о баге без проверки, что live `data/scripts` синхронизированы.
2. Строить правила только на внутренних документах без внешнего источника.
3. Использовать недокументированный API без явного fallback.
4. Привязывать допуск амбиций к префиксу строки вместо централизованного реестра.

## 8. Короткий чек-лист “Senior Gate” перед релизом

1. Есть внешние ссылки на ключевые правила?
2. Есть deploy/preflight подтверждение?
3. Есть лог-маркеры корректной загрузки?
4. Есть сценарий ручной приемки в игре?
5. Есть rollback-план?

Если хотя бы один ответ “нет” — изменение не готово.

## 9. Решения руководителя проекта (зафиксировано)

1. Тесты без preflight: **НЕ принимаются** (утверждено).
2. Поддержка старых сейвов после архитектурных изменений амбиций: **НЕ обязательна**.
3. Отдельный этап ROADMAP “стабилизация инженерного контура моддинга”:
   **добавить до новых сюжетных фич** (утверждено).
4. Пункт про single runtime-source (исключить dual-source): **утверждён (Вариант B)**.
   - Принцип: игра должна читать runtime-скрипты только из одного источника.
   - Для Noble Plus: runtime source = файлы репозитория `mod/mod_noble_plus/scripts/*`,
     подключённые в `Battle Brothers/data/scripts/*` через симлинки.
   - Любой fallback на копии допускается только временно и должен фиксироваться в логе preflight.

## 9.1 Техническая реализация решения B

1. One-time setup: `tools/setup_single_source_runtime.sh`
   - заменяет live-файлы в `data/scripts` на симлинки к репозиторию.
2. Перед каждым тестом: `tools/preflight_live.sh`
   - проверяет, что критичные файлы в `data/scripts` являются симлинками на repo source;
   - проверяет наличие обязательных лог-маркеров;
   - при несоответствии завершает работу с ошибкой.
