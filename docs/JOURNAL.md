# Noble Plus — Журнал разработки

## Как читать и вести этот журнал

Этот файл устроен в два слоя. Верхний слой — **краткая история проекта**: сжатое резюме всего, что было сделано до последних нескольких сессий. Нижний слой — **детальные записи последних 3-5 сессий** в полном виде.

Когда детальных записей накапливается больше пяти, агент в конце сессии обязан провести сжатие: самую старую детальную запись превратить в одну-две фразы в разделе "Краткая история", а её полный текст перенести в `JOURNAL_ARCHIVE.md`. Так журнал никогда не разрастается до размеров, засоряющих контекст.

Важное правило: в журнал попадают только **решения и причины**, а не просто список действий. "Создали файл X" — это не журнальная запись. "Решили использовать белого волка вместо вархаунда, потому что он сильнее и нарративно точнее соответствует образу родового зверя благородного дома" — это журнальная запись. Следуя этому правилу, журнал остаётся компактным и полезным даже спустя месяцы.

### Сессия 9 — 22 февраля 2026

**Что сделано.**

Проведён архитектурный рефакторинг системы амбиций: масштабируемое подавление, прогрессия между главами, нарратив battle_standard.

**Ключевые технические решения.**

Первое решение — **единый suppressor вместо N отдельных хуков**. Паттерн "один хук-файл на одну ванильную амбицию" не масштабируется: 22+ ванильных + DLC. Решение: `hooks/ambitions/noble_plus_vanilla_suppressor.nut` содержит список всех известных ванильных и применяет подавление через `foreach (p in vanillaAmbs) { ::mods_hookExactClass(p, fn) }`. Два старых файла (make_nobles_aware, allied_civilians) удалены.

Второе решение — **Score = 9999 для всех активных амбиций Главы I**. Прогрессивный Score 5–90 проигрывал конкуренцию ванильным. Теперь все активные Noble Plus амбиции имеют Score = 9999 — ванильные (Score 30–90) никогда не попадут в топ-4 пула.

Третье решение — **safety net в ambition_manager.setAmbition()**. Новый `hooks/ambitions/ambition_manager.nut`: если происхождение noble_plus и ID не начинается с `ambition.noble_plus.` — активация блокируется, `setDelay(1)`. Защита от неизвестных DLC-амбиций.

Четвёртое решение — **межглавная прогрессия через `::NoblePlus.tryFireChapterComplete`**. Добавлен хелпер в preload: принимает список флагов главы, флаг "глава завершена" и ID события. Каждая амбиция вызывает его при Done. Идемпотентен (проверяет _done_flag перед action). Создано событие `noble_plus_chapter1_complete_event.nut` — нарративный мост к Главе II.

Пятое решение — **нарративный SuccessText для battle_standard**. Ванильный текст про "кровожадное отребьё" заменён на текст про знамя Дома Локвуд ("серебряный волк без щита и без цепей — это армия претендента"). SuccessButtonText: "К делу." вместо ванильного.

**Статус после сессии.**

7 файлов изменено/создано, 2 удалено, задеплоено. Финальная верификация — за пользователем.

### Сессия 8 — 22 февраля 2026

**Что сделано.**

Диагностированы и устранены все 4 корневые причины неработающей системы амбиций (по скриншоту из сессии 7).

**Ключевые технические решения.**

Первое решение — **обнаружена архитектура VFS мода**. Папка `data/mod_noble_plus/` монтируется в VFS с префиксом `mod_noble_plus/`. Файлы в `hooks/` доступны как `mod_noble_plus/hooks/...`, но НЕ сканируются движком автоматически (движок сканирует только `scripts/`). Это корневая причина неработающих хуков — они были в нужном месте, но никогда не загружались.

Второе решение — **загрузка хуков через `::IO.enumerateFiles("mod_noble_plus/hooks")`**. Точный аналог паттерна Legends. Добавлен в `mods_queue` callback. Верифицировано: в логе нет предупреждений "never processed for hooks" для наших хуков.

Третье решение — **замена `Name` на `ButtonText`**. По исходникам Legends (`legend_roster_of_6_ambition.nut`): экран выбора читает `m.ButtonText`, а не `m.Name`. `Name` — нестандартное поле. Заменено во всех 3 кастомных амбициях.

Четвёртое решение — **явный `Score = 0` в хуках подавления**. Простой `return` без `Score = 0` оставляет ненулевой Score из `create()`, амбиция остаётся в пуле. Добавлен явный `this.m.Score = 0` перед `return`.

**Статус после сессии.**

Загрузка без ошибок, хуки применены. Финальная верификация — за пользователем. Ожидаемый результат: ровно 4 кастомных амбиции Noble Plus с правильными текстами, без ванильных.

### Сессия 6 — 22 февраля 2026

**Что сделано.**

Реализован этап 3.4 — система амбиций Главы I. Создано 3 кастомных файла амбиций + хук ванильной «Знамя отряда» + технический документ `docs/AMBITIONS_TECH.md`.

**Ключевые технические решения.**

Первое решение — **переход от последовательных к параллельным амбициям**. Первоначальный план предполагал гейтинг (1.2 открывается после 1.1, 1.3 после 1.2). В процессе уточнения требований установлено: стандартный механизм BB сам обеспечивает паттерн «4 → 3 → 2 → 1» — игрок выбирает из пула, а не получает по очереди. Все 4 амбиции доступны одновременно с первого дня.

Второе решение — **использование ванильной амбиции вместо кастомной**. Для «Знамя отряда» (4-я амбиция) использована ванильная `ambition.battle_standard` с хуком, добавляющим нарратив Noble Plus. Это сохраняет ванильную механику (2000 золота → штандарт-пика) и не дублирует код. Хук также ставит флаг `NoblePlus.Stage.OwnBanner.Done` при завершении.

Третье решение — **определение ID ванильной амбиции через код Legends**. Файлы ванильной игры скомпилированы в `.dat` (которые суть ZIP без .nut) — прочитать их напрямую нельзя. Но в хуке Legends `make_nobles_aware_ambition.nut` найдена прямая ссылка: `this.World.Ambitions.getAmbition("ambition.battle_standard")`. Отсюда выведен путь класса: `ambitions/ambitions/battle_standard_ambition`. Хук загрузился без предупреждения "never processed for hooks" → класс существует и применился.

Четвёртое решение — **подтверждение API `getBusinessReputation()`**. Тот же хук Legends использует `this.World.Assets.getBusinessReputation()` — это подтверждает правильность нашего API-вызова в `noble_plus_find_allies_ambition.nut`.

**Нерешённые вопросы (переходят в 3.5).**

Финальная верификация в игре (рассвет 2-го дня → экран выбора амбиции) не пройдена — её должен выполнить пользователь. Имена иконок (`ambition_roster_size`, `ambition_gold`, `ambition_renown`) не верифицированы — возможны ошибки при первом запуске. Авто-сканирование `data/scripts/ambitions/ambitions/` предполагается по аналогии с событиями, но не проверено экспериментально.

### Сессия 2 — 20 февраля 2026 (продолжение)

**Что обсуждали и решили.**

Принято решение вести документацию как персистентную память с механизмом сжатия журнала: детальные записи хранятся для последних 3-5 сессий, более старые сжимаются до одной-двух фраз в разделе "Краткая история" и архивируются в `JOURNAL_ARCHIVE.md`. Настроен проект в Claude Desktop — все будущие чаты по разработке создаются внутри него, что обеспечивает общий контекст и позволит в будущем настроить системный промпт проекта.

**Критические технические открытия этой сессии.**

Первое открытие — **структура виртуальной файловой системы BB**. Движок использует `data/` как корень виртуальной FS. ZIP-архивы модов виртуально "распаковываются" прямо в этот корень. Поэтому preload-скрипты должны лежать в `data/scripts/!mods_preload/`, а НЕ в `data/mod_noble_plus/scripts/!mods_preload/` — в последний путь движок не смотрит.

Второе открытие — **как движок регистрирует сценарии**. Движок сканирует `scripts/scenarios/world/` при раннем старте и создаёт экземпляры всех найденных классов — именно тогда вызывается `create()`. Это происходит ДО запуска очереди `mods_queue`. Если дополнительно вызвать `::include(...)` на тот же файл внутри `mods_queue`, файл загружается второй раз и переопределяет класс, но движок уже собрал список сценариев — второй вариант класса игнорируется. Вывод: файл сценария НЕ должен включаться через `::include` в preload, движок подхватывает его сам.

Третье открытие — **правильный родительский класс для кастомного сценария**. Мы сначала унаследовали от `disowned_noble_scenario` (DLC8), но это вызвало проблемы совместимости. Правильный базовый класс для кастомных сценариев Легенд — `starting_scenario`. Именно от него наследуют все кастомные сценарии в Legends (`legends_free_company_scenario`, `legends_mage_scenario` и др.).

Четвёртое открытие (незакрытое) — **ошибка MSU Settings API**. При попытке добавить панель настроек через `::NoblePlus.Mod.Settings` получили ошибку: `the index 'Settings' does not exist`. Это означает что либо API называется иначе, либо нужно сначала инициализировать объект настроек другим способом. Требует изучения документации MSU в следующей сессии.

**Статус на конец сессии.**

Мод регистрируется и загружается (`Noble Plus 0.1.0 загружен.` в логе). Класс сценария создаётся (`noble_plus_scenario::create() вызван.` в логе). Но игра падает при старте из-за ошибки в коде настроек MSU. Сценарий в меню выбора происхождений пока не проверен — нужно сначала убрать падающий код настроек.

**Что переходит в следующую сессию.**

Первый шаг: убрать или исправить блок `Settings` в preload-файле — он вызывает краш. Второй шаг: убедиться что сценарий "Дворянин +" появляется в меню выбора происхождений. Третий шаг: разобраться с правильным API MSU Settings и добавить панель настроек правильно.
